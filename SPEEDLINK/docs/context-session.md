# Speedlink Black Widow — коррекция осей через vJoy (контекст проекта)

> 🕹️ **Полный контекст проекта/сессии:** диагностика джойстика, коррекция осей и подача исправленного сигнала в vJoy, скрытие физического стика от MSFS.
> _Сгенерировано: 2026-06-05. Обновлено: 2026-06-05 (диагностика по USB-портам, замер дрожания, сброс калибровки реестра). Пользователь: Evgen Kamenskih._

## 1. Задача

Сделать GUI-программу диагностики манипуляторов и на её основе — программную коррекцию осей джойстика, которая:

1. Определяет все манипуляторы в системе (оси, кнопки, POV, VID/PID).
2. Проводит диагностику и выгружает её в лог (JSON + CSV).
3. По логу применяет коррекцию осей через **vJoy + vjoy_feeder.py**.
4. Работает после переподключения джойстика и перезагрузки ПК.
5. Прячет реальный физический джойстик от **MSFS 2020/2024** (устранение дубля).

## 3a. JoyDiag 2.0 — детали joy_diag.py (новая версия для всех осей)

`joy_diag.py` переписан до версии **2.0** и теперь работает со всеми осями манипулятора, а не только с X/Y.

### Возможности

- Поиск **ВСЕХ** манипуляторов (id 0..15) и их характеристик (VID/PID, оси, кнопки, POV, диапазоны).
- Живой просмотр всех 6 осей (`X, Y, Z, R, U, V`), кнопок и POV в реальном времени (~100 Гц). Синий маркер — сырое значение, зелёный — с коррекцией.
- Диагностика покоя (8с): центр (mean), шум (min/max/spread), σ, дрейф %, рекомендуемая deadzone.
- **Калибровка краёв** (12с): замер реальных min/max при полном отклонении каждой оси.
- Выгрузка **JSON** (с секцией `correction`) и **CSV** (сырые сэмплы покоя).
- **Запуск коррекции прямо из GUI** — кормит виртуальное vJoy-устройство скорректированными осями (через `pyvjoy`).

### Автогенерация параметров коррекции (из замера покоя)

- Если `|mean| > 0.9` и `spread < 0.02` → ось считается **нецентрируемой** (газ/слайдер): `type: throttle`, чистый проброс (`scale 1.0`).
- Иначе → `type: stick`:
  - `center_offset = -mean`
  - `deadzone = spread/2 + 2σ + 0.005` (рекомендуемая)
  - `scale_pos = 1/(1+c)`, `scale_neg = 1/(1-c)` (раздельный доводчик краёв; при достижимости < 0.05 берётся 1.0)
- Формула применения та же, что у фидера: `center_offset → deadzone-ремап → раздельный scale → invert → clamp[-1,1]`.

### Маппинг в vJoy и отличие от фидера

- Маппинг осей: `X→X, Y→Y, Z→Z, R→Rx, U→Ry, V→Rz`. Выход: `int(round((fixed+1)/2·32767))+1` (1..32768).
- Кнопки пробрасываются в vJoy (до 32).

> ⚠️ **Важно:** встроенная коррекция в `joy_diag.py` использует **pyvjoy** (`pip install pyvjoy` + драйвер vJoy) и **не** пробрасывает POV. Фоновый `vjoy_feeder.py` грузит `vJoyInterface.dll` напрямую (без pyvjoy) и пробрасывает кнопки **и** POV 1:1. Для постоянной работы в игре используется именно фидер; встроенная коррекция в GUI — для быстрой проверки профиля.

## 3b. Диагностика железа «почему уводит» + автоцентровка фидера

### joy_diag.py — раздел «Диагностика железа (почему уводит)»

- **Тест дрожания (60с)** — длинный замер покоя, разделяет причины увода:
  - высокий σ/шум → износ или грязь потенциометра (чистка/замена);
  - стабильный увод оси при малом шуме → механика/калибровка (лечится программно через `center_offset`);
  - заметный дрейф во времени → проседание USB-питания/порта (убрать USB-хаб, попробовать другой порт).
- **Тест переподключения** — снять центр, отключить и снова подключить девайс, нажать кнопку снова. Копит раунды в таблицу и считает разброс центра между ними.
- **Калибровка Windows** — читает из реестра сохранённую Windows-калибровку устройства. winmm отдаёт уже откалиброванные значения, поэтому:
  - если центр здесь смещён → виновата сохранённая калибровка (её нужно сбросить);
  - если ключа нет → увод чисто железный.
- **Сброс калибровки Windows** — удаляет пользовательскую ветку калибровки (HKCU без админа, HKLM с админом), после чего нужно переподключить USB.

### vjoy_feeder.py — автоцентровка при старте и каждом переподключении

**Ключевая проблема:** при подключении к разным USB-портам центровка стика может плавать — отсюда симптом «иногда уводит, иногда нет». Фиксированный центр из профиля для одного порта оказывается неверным для другого.

**Решение:** при каждом подключении джоя фидер ~1.2с слушает покой и заново вычисляет реальный центр стика. Центр больше не берётся фиксированным из профиля, а ловится заново под текущий порт.

**Причины плавающей центровки (почему зависит от порта/подключения):**

1. **Центр снимается в момент подключения / USB-enumeration.** Если в этот миг ручка не идеально в покое (пружина не устаканилась, лёгкое касание, наклон стола) — прошивка записывает смещённый центр. Передёрнул, когда ручка реально в покое → центр верный.
2. **Сохранённая калибровка Windows.** `winmm`/`joyGetPosEx` отдаёт уже откалиброванное Windows значение, а не сырое с железа. Windows держит калибровку в реестре, и кривой сохранённый центр гуляет от подключения к подключению (проверяется кнопкой «Калибровка Windows» в joy_diag.py).
3. **Питание по USB.** Некоторые модели берут опорное напряжение от Vcc USB; просадка на хабе/длинном кабеле слегка сдвигает «ноль» — отсюда зависимость от порта.

- **Защиты:** если стик трогают в момент замера (разброс > 6%) или центр подозрительно далёкий (> 0.6) — остаётся значение из профиля.
- **Флаги:** `--no-autocenter`, `--autocenter-secs 1.2`.

## 3c. Тест переподключения по портам + замер дрожания по портам (последняя итерация)

В joy_diag.py блок «Тест переподключения (поиск оптимального USB-порта)» реализован классом `ReconnectTest` и расширен замером дрожания по портам.

### Привязка к USB-порту (центр)

- Общее поле **«Метка порта»** + кнопки «Снять раунд», «Сброс таблицы», «Рекомендация».
- **«Снять раунд»:** быстрый замер центра X/Y (~1.5с) под текущей меткой → строка в таблице + лог `port_center_log.csv` (`timestamp, round, port_label, center_X, center_Y, sigma_X, sigma_Y, max_abs_drift`).
- **«Рекомендация»:** ранжирует порты по (увод центра + шум + разброс между раундами), помечает оптимальный, пишет `port_center_log_summary.txt`.

### Замер дрожания по порту (все 6 осей)

- Кнопка **«Замер дрожания (60с, все оси)»** — фиксированные 60с (константа `PORT_JITTER_SECS`), без отдельного поля ввода — по аналогии с кнопками «Замерить покой (8с)» и «Калибровка краёв (12с)».
- **Метка порта — общая** с тестом центра: по одному порту обе метрики (центр + дрожание) ложатся под одинаковой меткой и легко сопоставляются.
- **Лог отдельный:** `port_jitter_log.csv` (`timestamp, port_label, duration_s` + по 4 колонки `center_/sigma_/p2p_/drift_` на каждую из 6 осей X,Y,Z,R,U,V).
- **«Рекомендация по дрожанию»:** ранжирует порты по средним σ + p2p по всем осям, помечает самый тихий порт, пишет `port_jitter_log_summary.txt`.

### Ветки реестра с калибровкой и сброс

Калибровка джойстика в Windows хранится в двух ветках (в HKCU и/или HKLM):

- `...\\MediaProperties\\PrivateProperties\\Joystick\\OEM\\VID_07B5&PID_0317` (winmm/joyGetPosEx)
- `...\\MediaProperties\\PrivateProperties\\DirectInput\\VID_07B5&PID_0317` (DirectInput)

Полный префикс: `System\\CurrentControlSet\\Control\\MediaProperties\\PrivateProperties\\...`. Сброс = удалить подключ `VID_07B5&PID_0317` (HKCU — без админа, HKLM — с админом) + переподключить USB (без перезагрузки). Windows пересоздаст дефолтную калибровку при следующем подключении.

## 4. Профиль коррекции (joydiag_profile_final.json)

По последнему чистому замеру (2026-06-05, 524 сэмпла, edge_calibrated):

| Ось | Тип | center_offset | deadzone | scale_pos | scale_neg | Покой (mean) |
| --- | --- | --- | --- | --- | --- | --- |
| X (крен) | stick | −0.368449 | 0.0168 | 1.5834 | 0.7308 | +0.3684 (увод +36.8%) |
| Y (тангаж) | stick | +0.022201 | 0.0194 | 0.9783 | 1.0227 | −0.0222 (−2.2%) |
| Z (газ) | throttle | — | — | 1.0 (проброс) | — | стоял на полном ходу |
| R (руль) | stick | 0.0 | 0.005 | 1.0 | 1.0 | ~0 (идеально) |

`vjoy_target: 1`.

### Формула коррекции

- **stick/rudder:** `c = norm + center_offset` → deadzone-ремап `sign·(|c|−dz)/(1−dz)` → раздельный масштаб (`scale_pos` если c≥0, иначе `scale_neg`) → clamp[−1,1] → invert.
- **throttle/slider:** чистый проброс `clamp(norm,−1,1)` (без центровки), опц. invert.
- Маппинг в vJoy: `int(round((c+1)/2·32767))+1` (диапазон 1..32768).

### Проверка (логика, прогнано в тестах)

- X в покое → 0.000; полный вправо → +0.99; полный влево → −1.00.
- Y в покое → 0.000.
- Z: 0 → −1.0, середина → 0.0, полный → +1.0 (1:1, без центровки).
- R центр → 0.000.

## 5. Правильная последовательность применения

### Этап 0. Подготовка (один раз)

1. Папка `C:\\SPEEDLINK\\`, в неё все 5 файлов.
2. Установить: Python, vJoy (есть), HidHide.
3. В vJoyConf включить устройство #1: оси X/Y/Z/Rx, 8 кнопок, POV.

### Этап 1. Диагностика (по необходимости)

1. `python joy_diag.py` → выбрать физический Speedlink (VID 0x07B5).
2. «Замерить покой» (8с, руки убрать) → «Калибровка краёв» (12с, все оси до упоров, включая газ) → «Сохр. JSON».
3. JSON → пересобрать `joydiag_profile_final.json`.

### Этап 2. Проверка коррекции (до скрытия!)

1. `python vjoy_feeder.py`.
2. «Игровые устройства» → vJoy Device → крен в покое ровный, газ/руль на полный ход, кнопки/POV работают.
3. Остановить (Ctrl+C).

### Этап 3. Скрытие от MSFS (PowerShell от админа)

1. `setup_hidhide.ps1 -List` → найти `VID_07B5&PID_0317`, скопировать путь.
2. `setup_hidhide.ps1 -DevicePath "HID\\VID_07B5&PID_0317\\..."`.
3. Запустить фидер, проверить `vjoy_feeder.log`.

### Этап 4. Автозапуск

1. `setup_autostart.ps1` → старт сейчас и при каждом входе.

### Повседневно

- ПК включён → фидер сам поднялся → в MSFS только vJoy, без дубля, крен ровный.
- Переподключил USB → фидер сам подхватит (поиск по VID/PID).
- Смена коррекции → новый замер → заменить `joydiag_profile_final.json` → перезапуск фидера/задачи.

## 6. Как применять профиль

Просто положить `joydiag_profile_final.json` рядом с `vjoy_feeder.py` — фидер читает его при старте автоматически. Своё имя/путь можно задать аргументом: `python vjoy_feeder.py "C:\\путь\\profile.json"`. Если файла нет — берётся встроенный DEFAULT_PROFILE с теми же числами.

При смене профиля сам `vjoy_feeder.py` перекачивать НЕ нужно — он не зависит от конкретных чисел. Перекачка фидера нужна только при смене логики (новые оси, переход на DirectInput и т.п.).

## 7. Кнопка «Загрузить JSON» в joy_diag.py

Подтягивает ранее сохранённый профиль обратно в GUI (диапазоны, коррекция, vJoy target) — чтобы запустить коррекцию или посмотреть профиль без повторного замера. Обратная операция к «Сохр. JSON».

## 8. Известные нюансы и решения

- **SyntaxError: 'utf-8' codec can't decode byte 0xe4** — PowerShell/Notepad сохранил .py как ANSI/cp1251. Решение: использовать скачанный UTF-8 файл как есть, не пересохранять в ANSI, не создавать через `>`-редирект.
- **Y/Z/R «мёртвые» в логе** — артефакт: оси не двигали во время 8с покоя. Все оси живые (газ на dwZpos, полный 0–65535).
- **Калибровка краёв == caps [0,65535]** — у этого устройства полный ход и так заявлен в caps, поэтому edge-калибровка ничего нового не добавляет.
- **CSV содержит только фазу покоя** — для коррекции важны центр (из покоя) и края (из калибровки), это норма.
- **HidHide ↔ winmm:** фидер читает стик через winmm. Это работает, только пока `python.exe`/`pythonw.exe` в whitelist HidHide. Если после cloak в логе «джойстик не найден» — winmm не пробивает cloak на этой сборке, тогда перевести чтение фидера на DirectInput/raw HID.
